# MVP PLAN — Token Hunter

## 1. Cíl MVP

Postavit první verzi, která uživateli reálně odpoví:
- co je dnes nejvýhodnější,
- kde může ušetřit,
- co má smysl sledovat,
- a upozorní ho, když se objeví lepší deal.

## 2. Co musí MVP umět

### Must-have
- ingestovat první sadu klíčových zdrojů
- normalizovat provider/model/route/offer
- zobrazit home radar
- vrátit best buy pro pár základních use-case profilů
- ukázat deal detail s evidence trail
- umět watchlist a alerts

### Nice-to-have
- research digest
- hype/watch/relevant štítky
- budget profile

## 3. Doporučený MVP rozsah zdrojů

Začni úzkým coverage:
- hlavní API/providers a route katalogy,
- několik důvěryhodných discovery zdrojů,
- několik release/blog zdrojů pro AI/vibe coding.

Nezačínej širokým scrapingem všeho.

## 4. MVP use-cases

### UC1 — Coding cheap
Najdi nejlevnější rozumnou variantu na coding.

### UC2 — Experimentation free
Najdi free/free-ish možnosti pro testování.

### UC3 — Chat cheap
Najdi rozumný levný model na běžné použití.

### UC4 — Worth watching
Upozorni mě, když se objeví výhodnější route nebo nový relevantní deal.

## 5. MVP obrazovky

- Home Radar
- Compare Lite
- Deal Detail
- Watchlist
- Alerts Inbox
- Search Lite

## 6. MVP backend moduly

- source registry
- fetcher scheduler
- parser framework
- normalization pipeline
- verification layer
- scoring layer
- alerts engine
- API layer
- admin lite dashboard

## 7. MVP kvalita

Musí se měřit:
- stale deal rate,
- false alert rate,
- verified coverage rate,
- compare usefulness,
- time-to-verdict.

## 8. MVP kill criteria

Pokud se ukáže, že produkt nedokáže aspoň v úzké kategorii opakovaně dodat:
- jasnou úsporu,
- jasné rozhodnutí,
- nebo důvod vracet se,

je potřeba scope zúžit, ne rozšířit.

## 9. MVP success criteria

MVP uspěje, když:
- uživatel pochopí hodnotu během první relace,
- začne používat watchlist,
- alerty působí užitečně, ne otravně,
- compare opravdu vede k rozhodnutí,
- feed není jen přehlídka random novinek.
